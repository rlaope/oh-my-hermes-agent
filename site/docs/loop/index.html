Loop Engineering

In the AI ecosystem, after harness engineering, Loop Engineering is emerging as the next practical layer. The point is no longer just prompting. The point is giving a clear goal and designing the system that decides what to find, how to process it, when to hand off work, when to wait, and when to stop.

After harness engineering, Loop Engineering becomes the next operating surface for AI systems.

The goal is not to write a longer prompt. The goal is to define a durable objective, then let the system keep finding gaps, choosing the next step, preparing handoffs, evaluating feedback, and deciding when to continue, wait, or stop.

In OMH, ./loop gives Hermes Agent a visible loop contract for skills, plugins, connectors, subagents, worktrees, and external memory without pretending unobserved work already happened.

What the tick means

loop_runtime/v1 is a prepared queue, not execution evidence. It can name a worktree plan, subagent plan, connector intent, or executor handoff, but observed work must be recorded separately.

  • loop_engineering/v1 shows task discovery, distribution, execution, verification, and next-task decision state.
  • Automation, worktree, skill, connector, and subagent blocks stay visible as separate states.
  • Workflow patterns such as fan-out, adversarial verification, tournament, and triage batch are orchestration metadata, not dispatch evidence.
  • Cost policy keeps reads bounded, reuses stable schema scaffolds, and avoids extra verifier lanes unless evidence warrants them.
  • Prepared queue item is not worktree creation.
  • Prepared subagent plan is not subagent dispatch; subagent results should return status, summary, evidence refs, and next actions.
  • Prepared connector plan is not connector I/O.
  • External outcomes stay waiting until observed.

Start card to queue lifecycle

A chat wrapper can ask OMH for a loop_start_card/v1 before a loop artifact exists. After the user chooses a permission profile, Hermes can start the loop, tick it, show the prepared queue and loop engineering snapshot, render a handoff for the actionable item, then close that item as observed or blocked.

1 Start card

Explain the goal boundary and ask for permission profile.

2 Runtime queue

Prepare the next worktree, subagent, connector, or handoff plan.

3 Handoff

Render the selected queue item as executor-ready instructions.

4 Observe or block

Record evidence refs, or stop the queue item with a reason.
