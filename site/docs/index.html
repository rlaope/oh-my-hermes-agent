Documentation

Use OMH as a local contract layer for Hermes.

OMH gives Hermes installed skill guidance for turning natural language into deterministic routing, clarification, planning, status narration, or Codex-ready coding handoffs. Local backend contracts and schema-versioned artifacts support wrappers and operators without becoming the normal user UX.

Install

Install through Hermes skill taps when available. This keeps the main experience inside Hermes.

hermes skills tap add rlaope/oh-my-hermes-agent hermes skills install oh-my-hermes

Use the OMH bootstrap path when you need generated managed skills, local doctor checks, or wrapper/backend commands in the same runtime.

curl -fsSL https://raw.githubusercontent.com/rlaope/oh-my-hermes-agent/main/install.sh | sh omh setup omh doctor

Stable installs are pinned explicitly with a released tag.

OMH_CHANNEL=stable OMH_VERSION=<version> \ sh -c "$(curl -fsSL https://raw.githubusercontent.com/rlaope/oh-my-hermes-agent/v<version>/install.sh)"

Chat wrappers

Wrappers own platform-specific work: Discord or Slack auth, incoming events, buttons, threads, edits, and notifications. Hermes owns the chat experience; OMH owns the deterministic backend response contract when a wrapper needs renderable state.

omh chat interact --source discord "risky refactor" omh chat interact --source slack "diagnose installation health"

These are adapter backend calls. A response can be an answer, a clarification request, a plan, a status card, or a prepared handoff. The wrapper should render that response natively instead of asking users to memorize CLI commands.

Hermes Agent Integration Runbook

Operators should treat OMH as the local contract layer consumed by a Hermes-agent wrapper, not as a transport bot. The runbook explains which fields to render, which owner is responsible for each transition, and which claims are still not evidence.

  • Wrapper owns platform events, buttons, threads, edits, and notifications.
  • OMH owns deterministic contracts, fixture examples, prepared handoffs, and status boundaries.
  • Codex-like executors own implementation evidence only after dispatch is observed.

Read the source runbook at docs/HERMES_AGENT_INTEGRATION_RUNBOOK.md and validate the matching golden fixture at examples/wrapper-golden/hermes-agent-integration.json.

Local demo

Use the deterministic orchestration demo to inspect the full wrapper path before building a real adapter.

omh demo orchestration uv run python examples/discord-adapter-shim.py uv run python examples/slack-adapter-shim.py

The demo and shims are fixture-backed. They do not call bot SDKs, LLM APIs, network services, or Hermes internals.

Architecture at a glance

OMH starts as installed Hermes skills. Wrapper backends can consume deterministic local contracts while Hermes stays responsible for conversation, planning, and status narration.

User message Hermes, Discord, Slack, hosted chat
->
Hermes Agent Clarify, research, plan, narrate
->
Wrapper adapter Buttons, threads, edits
->
OMH local contract layer Playbook, route, plan, handoff, status
Installed OMH skills Tap install or omh setup
Codex-like executor Implementation after dispatch
Runtime artifacts Observed evidence and status cards

Prepared handoff is only a ready-to-dispatch state. Execution, verification, review, CI, and merge readiness stay separate until runtime evidence is observed.

Playbooks

Playbooks sit above individual skills. Hermes users should not run them directly; wrappers and maintainers can inspect situation-level pipelines for source-backed research, deep interview to plan, safe coding handoff, local pipeline buildout, and release-readiness review.

omh playbook list omh playbook inspect safe-feature-change omh playbook recommend "research latest official sources"

Use playbooks when wrappers need to know which stages stay with Hermes, which stages become executor handoffs, and which claims are not evidence until runtime observations exist.

Contracts

chat_interaction/v1

Input envelope for natural messages, source metadata, thread context, and user intent.

chat_response/v1

Output envelope for wrapper-facing text, action hints, status snippets, and handoff prompts.

status_card/v1

Readable status summary for plan, dispatch, implementation, review, CI, and merge readiness.

harness_progress/v1

Evidence-based progress ladder that advances only when required observations are present.

Delegation-first coding

OMH does not hide a coding executor inside Hermes. Coding-heavy requests are shaped into prepared handoffs for Codex-like agents with scope, acceptance criteria, verification commands, review expectations, Codex $skill invocation guidance, and merge-readiness evidence.

  • Prepared handoff means "ready to dispatch", not "work completed".
  • Observed implementation requires linked runtime evidence.
  • Review, CI, and merge-readiness require explicit observed records.

Quality gates

OMH keeps skill and harness metadata inspectable through the CLI so wrappers and operators can validate contract coverage before depending on it.

omh docs workflows --json omh harness list omh harness inspect planning omh harness validate

Boundaries

OMH is deterministic local tooling. It does not patch Hermes core, run platform bots, call LLM APIs, create network integrations inside core commands, or launch Codex by itself. Those responsibilities stay explicit at the wrapper, executor, or operator layer.

For source files, release notes, and contribution workflow, use the repository after you understand the public contract from this page.
